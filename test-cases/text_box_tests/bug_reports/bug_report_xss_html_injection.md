# [Bug]: Уязвимость к HTML-инъекциям (XSS) в текстовых полях формы

> **📌 Note for Reviewers (Дисклеймер):** > *Данный баг-репорт является симуляцией (Simulated Report), созданной специально для портфолио. В реальности форма на DemoQA корректно санитизирует ввод. Этот документ демонстрирует мои навыки оформления критических уязвимостей безопасности (Stored XSS) и понимание стандартов OWASP в случае их обнаружения на реальном проекте.*

**Environment:** - OS: Windows 11 / macOS 14
- Browser: Chrome 122.0
- URL: https://demoqa.com/text-box

## Описание (Description)
В текстовых полях формы (Full Name, Current Address, Permanent Address) отсутствует санитизация (экранирование) пользовательского ввода. Введенные HTML-теги не преобразуются в безопасные сущности (строки), а рендерятся браузером как исполняемый код. Это приводит к поломке верстки (UI Break) и создает критическую уязвимость для Cross-Site Scripting (XSS) атак, что недопустимо по стандартам безопасности OWASP.

## Шаги для воспроизведения (Steps to Reproduce)
1. Открыть страницу https://demoqa.com/text-box
2. В поле **Full Name** ввести: `<h1>HACKED</h1>`
3. В поле **Email** ввести: `test@mail.com`
4. В поле **Current Address** ввести: `<button style="width: 100vw; height: 100vh;">Click Me</button>`
5. В поле **Permanent Address** ввести: `<script>alert('XSS Vulnerability')</script>`
6. Нажать кнопку **Submit**

## Ожидаемый результат (Expected Result)
Форма отправляется. Введенные спецсимволы и теги корректно экранируются (например, преобразуются в `&lt;h1&gt;HACKED&lt;/h1&gt;`). Под формой отображается простой текст без применения стилей, HTML-разметки или выполнения JavaScript.

## Фактический результат (Actual Result)
Введенные HTML-теги рендерятся браузером как часть DOM-дерева:
- Имя отображается огромным заголовком H1.
- В адресе появляется гигантская кнопка на весь экран, ломающая интерфейс.
- *Опционально:* Скрипт в `Permanent Address` пытается выполниться.

## Дополнительная информация (Additional Context)
