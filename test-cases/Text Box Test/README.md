# Тестовая документация: Форма Text Box (DemoQA)

## 📌 Вводные данные и допущения (Assumptions)

Поскольку для данного проекта отсутствует официальная спецификация (ТЗ), тест-дизайн выполнен на основе реверс-инжиниринга текущего функционала с применением **общепринятых индустриальных стандартов веб-разработки** (Heuristics & Web Standards). 

Ожидаемые результаты в тест-кейсах опираются на следующие технические правила нормального поведения системы:

1. **Ограничения базы данных (Limit & Boundary):**
   - Поля `Full Name` и `Email` ограничены длиной в 255 символов (стандарт для `VARCHAR(255)`).
   - Поля адресов (`Current Address`, `Permanent Address`) ограничены длиной в 1000 символов.
   - *Ожидание:* При попытке ввести текст, превышающий лимиты, система должна ограничивать ввод на уровне UI или возвращать ошибку валидации, не допуская отправки тяжелых запросов на бэкенд.

2. **Безопасность (Security):**
   - Вводимые данные санитизируются. Любые HTML-теги, скрипты или SQL-команды экранируются и обрабатываются как обычный текст для предотвращения XSS-уязвимостей и SQL-инъекций.

3. **Валидация форматов (Data Validation):**
   - Поле `Email` жестко валидируется: принимает только адреса, содержащие имя ящика, символ `@` и корректную доменную зону верхнего уровня.

4. **Очистка данных (Sanitization):**
   - Система применяет функцию обрезки пробелов (`trim()`). Ввод, состоящий исключительно из пробелов, приравнивается к пустому значению. Лишние пробелы в начале и конце строк удаляются перед сохранением.
